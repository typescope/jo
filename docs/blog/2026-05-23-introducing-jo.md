# Introducing Jo — Secure Programming for the AI Era

We are excited to introduce **Jo**, a statically typed functional programming language built around a simple idea: security and correctness should be the default, not an afterthought.

## Why Jo?

Modern software runs in increasingly hostile environments. AI agents autonomously execute code, microservices call each other across trust boundaries, and the blast radius of a single compromised component can be enormous. Yet most languages still require developers to opt *into* security rather than opt *out* of it.

Jo flips this model. By baking capability-oriented programming, an effect system, and a rich type system directly into the language, Jo makes it structurally difficult to write code that violates security invariants — even when that code is generated by an AI.

## Core Features

### Capability-Oriented Programming

In Jo, access to system resources — the filesystem, network, environment variables — is controlled through *capabilities* passed explicitly as parameters. A function that cannot receive a `FileSystem` capability literally cannot read or write files. There is no ambient global access to exploit.

```jo
def readConfig(path: String)(using fs: FileSystem): Config =
  fs.read(path) |> parseConfig
```

### Effect System

Jo tracks *effects* at the type level. A function's signature tells you not just what it returns, but what side effects it may perform. This makes it possible for tools — and humans — to reason about code behavior without reading implementations.

### Union Types and Pattern Matching

Jo's union types let you model domains precisely, and exhaustive pattern matching ensures every case is handled:

```jo
type Shape = Circle | Rectangle | Triangle

def area(s: Shape): Float = s match
  case Circle(r)        => Math.PI * r * r
  case Rectangle(w, h)  => w * h
  case Triangle(b, h)   => 0.5 * b * h
```

### Multiple Compilation Backends

Jo compiles to JavaScript, Ruby, Python, and native Linux/x86 code — one language, any target.
