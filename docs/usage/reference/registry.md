# Registry

The Jo package registry has two separate concerns:

- package registration metadata
- published release metadata

Package registration is a manual pull-request workflow. This keeps package ownership and publication sources explicit and reviewable in the registry repository.

## For Library Authors

If you want to publish a library, the registry model is:

1. package the library with `jo package`
2. register the package manually in the registry repository the first time you publish it
3. publish release artifacts through the configured publication source
4. wait for the registry daemon to detect the new release

The important point is:

- registration is manual and reviewed
- release discovery is automatic
- a version is considered published only after it appears in the registry release log

## Registration Workflow

Registration is manual:

1. package the library and confirm its namespace is correct
2. add the package registration metadata to the registry repository
3. open a pull request for that registration
4. after approval, package ownership is recorded there

This is intentionally manual because package ownership requires review.

## Registration Metadata

Each package has one registration file in `registry/`:

```toml
namespace = "agentapi.core"
name = "agent-api"
repo = "https://github.com/typescope/agent-api"
owners = ["alice@example.com"]

[publish]
github = "typescope/agent-api"
```

It defines:

- package name
- namespace
- source repository
- owners
- publication source

Because ownership policy is social and administrative, it is not automated by the tool.

The publication source is declared in a `[publish]` section. For now, the only supported source is GitHub Releases.

```toml
namespace = "agentapi.core"
name = "agent-api"
repo = "https://github.com/typescope/agent-api"
owners = ["alice@example.com"]

[publish]
github = "typescope/agent-api"
```

## Release Metadata

Each package has one append-only JSONL file in `releases/`:

```jsonl
{"version":"1.0.0","url":"https://.../agent-api-v1.0.0.joy","sha512":"...","source_url":"https://.../agent-api-v1.0.0-sources.zip","source_sha512":"..."}
{"version":"1.1.0","url":"https://.../agent-api-v1.1.0.joy","sha512":"...","source_url":"https://.../agent-api-v1.1.0-sources.zip","source_sha512":"..."}
```

Each line is one published release.

Required fields:

- `version`
- `url`
- `sha512`

Standard optional fields:

- `source_url`
- `source_sha512`

The dependency resolver only needs the `.joy` artifact fields. Source artifact fields exist for auditing and future tooling.

These files are generated by a registry daemon. They are not edited manually.

## Supported Publication Protocol

For now, Jo supports only one publication source:

- GitHub Releases, declared by `[publish].github`

The registry daemon scans the configured GitHub repository and accepts a release only when it matches the registry contract.

For version `1.2.0`, expected artifact names are:

- `<name>-v1.2.0.joy`
- `<name>-v1.2.0-sources.zip`

The daemon downloads the `.joy` artifact and computes `sha512` itself before writing the canonical release entry. If a source archive is present, it computes `source_sha512` too.

GitHub is treated as the publication transport, not as the canonical source of truth. The canonical source of truth for clients is the generated `releases/.../<package>.jsonl` file.

## Client Access Pattern

Clients resolve packages by exact package name. They do not need a single global database.

The published package endpoint is:

```text
pkg.jo-lang.org/<package-name>.jsonl
```

That endpoint is a flat, client-facing projection of the release metadata for one package.

## Repository Layout

The registry repository is organized as:

```text
registry/
releases/
README.md
```

`registry/` stores package identity and ownership metadata.

`releases/` stores published release metadata.

Both trees use the same directory layout:

```text
<shard>/<top-level-namespace>/<package-file>
```

where:

- `shard` is the first two letters of the top-level namespace
- `top-level-namespace` is the full first namespace segment

Example:

```text
registry/
  ag/
    agentapi/
      agent-api.toml
  jo/
    jo/
      jo-http.toml
  pa/
    parsing/
      parsing-lexer.toml
      parsing-parser.toml
  sq/
    sql/
      jo-sqlite.toml

releases/
  ag/
    agentapi/
      agent-api.jsonl
  jo/
    jo/
      jo-http.jsonl
```
