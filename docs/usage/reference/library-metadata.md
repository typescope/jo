# Library Metadata

Each `.joy` archive contains a `meta.toml` which contains metadata about the package. The file is generated by the compiler (`jo package`).

::: info
Library metadata (`meta.toml`) and build spec (`jo.toml`) serve different purposes: the build spec is instructions for *producing* an artifact; `meta.toml` describes the *produced* artifact for consumers. `jo package` derives `meta.toml` from the build spec.
:::
## Fields

| Field         | Type            | Description |
|---------------|-----------------|-------------|
| `name`        | string          | Package name. Copied from `[package].name`. |
| `jo`          | string          | Required Jo compatibility line. Copied from `jo`. Mandatory. |
| `version`     | string          | Semantic version. Copied from `[package].version`. |
| `namespace`   | string          | Namespace declared in the source files. Determined by the compiler. |
| `ffi`         | string          | FFI platform: `"none"`, `"python"`, or `"ruby"`. Computed by the compiler. |
| `description` | string          | Copied from `[package].description`. |
| `authors`     | array of string | Copied from `[package].authors`. |
| `license`     | string          | Copied from `[package].license`. |
| `homepage`    | string          | Copied from `[package].homepage`. |
| `keywords`    | array of string | Copied from `[package].keywords`. |
| `dependencies`| table           | Resolved dependency ranges, derived from `[main.dependencies]`. |

## The `ffi` Field

`ffi` is always present and computed by `jo package` from direct FFI usage in the source and from transitive dependencies. It is never set by the author in `meta.toml`.

The author may optionally declare `ffi` in `jo.toml` as an assertion. If declared, `jo package` verifies the computed value matches — a mismatch is a build error. This is useful for libraries that want to guarantee platform-independence:

```toml
[package]
ffi = "none"    # build error if any dep introduces FFI
```

`ffi` is **contagious**: if any dependency has `ffi != "none"`, the package inherits that value. Two dependencies with conflicting `ffi` values is a build error.

## Namespace-to-Directory Mapping

Dots in the namespace become directory separators. The compiler locates `.sast` files under the path derived from the declared namespace:

```
namespace = "agentapi"       →  agentapi/
namespace = "parsing.lexer"  →  parsing/lexer/
```

The compiler enforces this at both pack time (`jo package`) and unpack time (`jo build`). A `.joy` whose directory layout does not match its declared namespace is rejected.

## Examples

### Example meta.toml

```toml
name        = "agent-api"
jo          = "1.0"
version     = "1.0.0"
namespace   = "agentapi"
ffi         = "none"
description = "Sandbox agent framework API"
authors     = ["Alice <alice@example.com>"]
license     = "MIT"
homepage    = "https://github.com/alice/agent-api"
keywords    = ["agent", "framework"]

[dependencies]
mustache = "1.0"
```

### Example archive layout

```
agent-api-v1.2.0.joy
  meta.toml
  agentapi/
    AgentAPI.sast
    QueryDSL.sast

agent-runtime-python-v1.0.0.joy
  meta.toml
  agentruntime/
    AgentRuntime.sast
```
